CodexBridge 本地微信接入：把 Codex Agent 放进 ChatOps 链路

一次 Windows 本地微信接入实践：修复依赖、二维码登录、iLink 长轮询、Codex CLI 路由和运行手册，让微信 ClawBot 能触发本地 Agent 工作流。

背景

CodexBridge 的目标是把 Codex 接入不同聊天平台，让本地 Agent 不只停留在终端里。此次实践聚焦 Windows 本地微信链路：从仓库启动 WeChat + Codex 桥接服务，并在微信 ClawBot 中让 /status 和 /helps 成功回复。

这类项目的关键不在于单次模型调用，而在于把登录、消息接收、命令路由、出站回复和本地运行状态串成稳定链路。

端到端结果

本机已经完成一次完整接入：

  • ClawBot iLink 登录成功，账号状态保存到用户级状态目录。
  • weixin:serve 能连接 iLink getupdates 并持续长轮询。
  • 微信侧发送 /status 和 /helps 已确认成功回复。
  • 默认工作目录使用 C:/Users/yourskenny/Documents/Codex，便于从微信进入本地 Codex 工作流。

用户级状态目录不随仓库迁移，token 和账号状态保存在 C:/Users/yourskenny/.codexbridge，不提交到 Git。

本地化改造

克隆后不能直接跑通，原因是多个 Windows 和 iLink 现实问题叠加：

  • lockfile 与 package.json 不同步，导致 npm ci 缺依赖。
  • ffmpeg-static 安装脚本在本机 Node 默认 CA 下证书校验失败。
  • iLink 返回的 qrcode_img_content 实际是登录 URL，旧逻辑会把 HTML 页面保存成 .png。
  • silk-wasm 是可选动态依赖，缺少类型声明会让 JS 类型检查失败。
  • Windows 非管理员环境创建 symlink 会 EPERM，测试需要条件化。
  • 插件分类数量相同时依赖 locale 排序，导致测试顺序不稳定。
  • 本地运行日志和二维码文件需要隔离到 .codexbridge-run/。

这些改造把项目从“能在某个环境临时跑起来”推进到“可以在这台 Windows 机器上复现、验证和交接”。

验收链路

我把验证拆成几层：

  1. 安装与静态检查：npm ci、npm run typecheck、npm run typecheck:js、npm test。
  2. 登录：npm run weixin:login -- --state-dir C:/Users/yourskenny/.codexbridge。
  3. 服务：设置 Codex CLI 路径和微信策略后运行 npm run weixin:serve。
  4. 微信侧验收：向 ClawBot 发送 /status 和 /helps。
  5. 排障：观察 .codexbridge-run/weixin-serve.out.log 中的 getupdates、messageCount 和出站发送记录。

这样可以判断失败发生在登录、收消息、命令路由、Codex 执行还是出站回复，而不是在一个黑盒里猜。

工程收获

这个项目体现了 Agent 工程中很实际的一面：模型能力之外，真实可用性依赖运行环境、状态管理、权限、日志、测试稳定性和操作文档。

把 Codex 接入微信后，微信不只是聊天入口，而可以成为本地 Agent 工作流的控制面。后续可以继续把它接入 GSD、AstrBot 或其他本地工具链，形成一个更统一的 ChatOps Agent 系统。
