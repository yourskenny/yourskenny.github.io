本地 Agent 桥接、Windows 适配与端到端验收

CodexBridge：Windows 本地微信到 Codex 的 Agent 桥接

把微信 ClawBot、iLink 长轮询和 Codex CLI 串成可本地运行的 ChatOps Agent 桥接服务，支持登录、消息接收、slash command 路由和运行手册化交付。

独立项目 C:/coding/CodexBridge
AgentChatOpsWeChatCodexTypeScriptWindows

项目简介

CodexBridge 是一个以 Codex 为中心的多聊天平台桥接项目。我在 Windows 本地完成了微信接入链路：通过 ClawBot iLink 登录微信账号，使用 getupdates 长轮询接收消息，再把微信侧的 /status、/helps 等命令路由到 CodexBridge 服务中执行并返回回复。

这个项目的价值不是单纯把一个 bot 跑起来，而是把 AI Agent 从命令行迁移到日常 IM 场景，让微信成为 Codex 工作流的 ChatOps 入口。

技术栈

  • TypeScript / Node.js
  • Codex CLI
  • OpenAI Agents SDK
  • WeChat ClawBot / iLink
  • QRCode 登录、长轮询消息收发
  • Windows PowerShell、本地状态目录与后台服务脚本

我的工作

我负责把项目迁移到 C:/coding/CodexBridge 后完成 Windows 本地可复现运行，包括依赖安装、类型检查、测试修复、二维码登录修复、微信消息链路验收和运行手册整理。

端到端验证结果包括：

  • ClawBot iLink 登录成功，账号状态保存到用户级状态目录。
  • weixin:serve 能连接 iLink getupdates 并持续长轮询。
  • 微信侧发送 /status 和 /helps 已确认能收到回复。
  • 默认工作目录指向 C:/Users/yourskenny/Documents/Codex，便于从微信侧触发本地 Codex 工作流。

方法与实现

项目把微信账号状态放在用户级目录 C:/Users/yourskenny/.codexbridge，仓库本身只保留代码和运行手册，避免把 token 或登录状态提交进 Git。

本地化改造重点包括：

  • 修复 lockfile 与 package.json 不同步导致的 npm ci 失败。
  • 通过 .npmrc 让 npm 生命周期使用系统 CA，解决 ffmpeg-static 安装证书问题。
  • 将 iLink 返回的登录 URL 渲染成真实 PNG 二维码，而不是把 HTML 页面误保存为图片。
  • 为可选语音转码依赖补充本地类型声明，保证 JS 类型检查通过。
  • 对 Windows symlink 测试做条件化处理，避免非管理员环境下 EPERM 误伤测试。
  • 固定插件分类排序，消除 locale 排序带来的测试不稳定。
  • 把 .codexbridge-run/ 作为本地运行产物目录，隔离日志和二维码辅助文件。

难点与解决

这个项目的难点来自多层系统边界：微信登录状态、iLink 长轮询、Codex CLI、Node 依赖安装、Windows 权限和本地运行产物都可能让链路中断。

我把验证拆成几个可观察层：

  1. npm ci、typecheck、typecheck:js、test 保证代码基础健康。
  2. weixin:login 验证二维码登录和 token 保存。
  3. weixin:serve 验证长轮询收消息。
  4. 微信侧 /status、/helps 验证命令路由和出站回复。
  5. 后台服务日志落到 .codexbridge-run/，用于后续排障。

这种分层验证让问题能被定位到具体环节，而不是把所有失败都归因于“微信没通”或“Agent 没响应”。

复盘

CodexBridge 适合作为“AI Agent 进入真实通信入口”的项目展示。它证明了我能处理模型调用之外的工程问题：登录态、后台服务、运行手册、Windows 环境差异、测试稳定性和端到端验收。

后续可以继续扩展为多平台 ChatOps 控制台，例如把微信、NapCat/AstrBot 和 Codex/GSD 工作流统一到一个本地 Agent 调度入口。
