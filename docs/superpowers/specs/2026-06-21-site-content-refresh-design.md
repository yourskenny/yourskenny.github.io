# 个人网站内容更新设计

## 背景

网站当前已经有项目、博客、课程归档、研究方向和关于页。用户这次的目标不是只写博客，而是把最近一段时间的项目、面试交付、复习工作流和履历材料整理成适合公开展示的网站内容。

本次更新面向求职和实习展示，优先突出可运行项目、工程闭环、验证意识、独立推进能力和清晰事实边界。

## 目标读者

- 实习面试官和招聘方：希望快速判断项目是否真实、是否能运行、是否有工程质量。
- 导师或科研合作者：希望看到问题意识、方法抽象和长期维护能力。
- 未来的自己：希望能从网站反查每段经历对应的公开叙事和证据边界。

## 内容放置原则

内容先按材料性质分类，不预设必须写成博客。

| 内容类型 | 放置位置 | 判断标准 |
| --- | --- | --- |
| 完整项目或可运行交付 | 项目页 | 有明确目标、实现、技术栈、产物和验证方式 |
| 面试交付项目 | 面试项目板块 | 为面试或试标任务完成，能体现短周期交付和工程判断 |
| 方法论、复盘、踩坑 | 博客 | 重点是解释方法、决策、失败原因或可迁移经验 |
| 课程资料集合 | 课程归档 | 主要价值是学习轨迹和课程实验覆盖 |
| 个人定位和入口 | 首页 / 关于页 / 研究页 | 用于串联代表项目、奖项、方向和联系入口 |

## 信息架构调整

### 面试项目

新增独立面试项目板块，用于收纳面试期间完成的短周期交付。

首批内容：

- Yahaha Agent Arcade
  - 类型：面试项目
  - 来源：`C:\coding\YAHAHA`
  - 公开重点：三天 MVP、creator prompt 到 playable bundle 的闭环、Agent Harness、对象存储、Dashboard、质量门禁、测试与部署文档。
  - 不夸大：不写成生产 SaaS；明确 demo auth、in-process tasks、local deterministic agent 等边界。

- T3-Agent Rubric 试标交付包
  - 类型：面试项目
  - 来源：`C:\coding\rubric`
  - 公开重点：规则理解、先做题再写 rubric、约束台账、答案证据台账、自检清单、交付包组织。
  - 脱敏要求：不公开外部平台 IP、原始规则 PDF、视频摘要原文、题包细节或面试方敏感材料。

实现上可复用 `projects` 内容集合，并扩展 `category` 枚举加入 `面试项目`。站点可以新增 `/interview/` 页面，也可以在项目页中单独分组。为满足“单独安排一个位置”，推荐新增 `/interview/`，并从导航和项目页入口链接过去。

### 项目作品集

新增或更新适合求职展示的项目页。

新增候选：

- LCFC NL2SQL 智能问数系统
  - 类型：竞赛项目 / Agent 工程
  - 公开重点：自然语言解析、schema 检索、结构化 IR、受控 SQL compiler、结果校验、benchmark 和提交包生成。
  - 边界：比赛结果未确认前，不写获奖、排名或未公开分数。

- AI PR Review Assistant
  - 类型：开发者工具 / AI 工程
  - 公开重点：PR metadata 和 diff 获取、新增行号绑定、低噪声 finding、上下文预算、Markdown / JSON / SARIF 输出、Dashboard 和 GitHub comment 能力。
  - 边界：比赛结果未确认前，不写获奖。

- EchoGrid
  - 类型：竞赛项目 / 独立游戏开发
  - 公开重点：玩法设计、交互反馈、关卡体验、从原型到可提交版本的闭环。
  - 边界：比赛结果未确认前，不写获奖或入围。

- 商贸营销智能体
  - 类型：研究原型 / Agent 工程
  - 公开重点：业务流程拆解、Agent 能力矩阵、上下文管理、工具编排、可解释推理记录、RESTful API 接入思路。
  - 脱敏要求：不公开原始需求文档全文或具体业务敏感信息。

更新候选：

- FEO 工业视觉项目
  - 已有项目页，不新建重复内容。
  - 可补充已确认的中国大学生计算机设计大赛省级二等奖事实。
  - 避免写无证据的生产部署指标或线上效果。

### 博客

博客只承载复盘、方法论和可迁移经验，不替代项目页。

首批候选：

- 受控 NL2SQL 不等于让模型直接写 SQL
  - 来源：LCFC
  - 类型：AI 工程实践
  - 目标：解释结构化 IR、受控 compiler、结果校验和 benchmark 为什么重要。

- 从 PR URL 到低噪声代码评审
  - 来源：AI PR Review Assistant
  - 类型：AI 工程实践
  - 目标：解释 finding 结构、上下文预算、报告格式和降级机制。

- 从聊天式复习到可追踪复习工作流
  - 来源：`yourskenny/junior-spring-exam-review`
  - 类型：项目复盘或 AI 工程实践
  - 目标：把模块拆分、资料权重、进度状态、错题表、方法卡和会话交接整理成可复用学习工作流。
  - 放置：只作为博客，不单独新增复习工作流栏目。

- YOLO-FD 失败复盘
  - 来源：`C:\coding\YOLO-FD`
  - 类型：项目复盘
  - 目标：诚实记录投入、调参、失败原因和停止继续投入的判断。
  - 边界：不包装成强性能成果。

### 首页、关于页和研究页

首页需要把代表项目调整为更符合当前求职目标的组合：

- Agent 工程主线：LCFC、AI PR Review、Yahaha、标准化 Agent 工作流。
- 成熟落地佐证：FEO。
- 特殊板块入口：面试项目。

关于页需要更新：

- 当前定位：AI Agent / AI 工程方向优先，兼具 CV、多模态和游戏开发经历。
- 已确认奖项：FEO 省级二等奖、CCF 量子计算编程挑战赛三等奖。
- 面试项目和公开网站作为工程交付能力补充。

研究页可轻量更新：

- 增加 Agent / NL2SQL / 可信 AI 工程方向。
- 保留 CV 和多模态，但不再让研究页只读成 CV 方向。

## 公开边界

本次所有内容必须遵守这些规则：

- 未公布结果的比赛不写获奖、入围、排名。
- 非主导项目明确写“参与开发”或降低展示权重。
- 面试材料只写方法和交付结构，不公开对方原始资料、内部链接、平台地址或敏感题包。
- 私有仓库项目只写可公开的架构、流程和贡献，不泄露私有数据、账号、提交包或评测细节。
- FEO 只写已确认的省级二等奖，不写国赛二等奖或无证据指标。
- YOLO-FD 作为失败复盘，不包装成高性能检测成果。

## 实施顺序

1. 扩展内容结构：增加 `面试项目` 分类和 `/interview/` 页面。
2. 新增面试项目内容：Yahaha 和 Rubric。
3. 新增核心项目页：LCFC、AI PR Review、EchoGrid、商贸营销智能体。
4. 更新已有 FEO 项目页。
5. 新增博客：junior-spring-exam-review 复习工作流，随后按优先级写 LCFC / AI PR Review 工程复盘。
6. 更新首页、项目页、关于页和研究页入口。
7. 运行内容校验、测试和构建。
8. 按既有发布流程推送 `source` 并部署 `main`。

## 验证方式

- `npm test` 验证站点内容结构和生成脚本。
- `npm run build` 验证 Astro 内容集合、路由和页面构建。
- 本地预览检查 `/interview/`、`/projects/`、`/blog/`、`/about/` 和 `/research/`。
- 发布后用 `curl` 检查 `https://yourskenny.github.io/` 和 `https://yourskenny.top/` 的新增页面。

## 不在本次范围

- 不重设计整站视觉风格。
- 不新增复杂 CMS 或后台。
- 不公开未脱敏的面试材料和比赛提交包。
- 不把 junior-spring-exam-review 做成独立栏目。
- 不为每个本地项目都强行写页面，先聚焦能服务求职展示的内容。
